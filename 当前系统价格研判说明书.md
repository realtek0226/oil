# 当前系统价格研判说明书

## 1. 研判对象

当前系统主要研判两个对象：

1. 山东 92#汽油市场价
   - 输出趋势、点位、区间、研判可靠度、核心驱动、经营建议。
   - 支持周期：D1、D3、W1、M1。

2. 其他区域与山东的 92#汽油价差
   - 当前已覆盖华东-山东、华北-山东、华南-山东、华中-山东、西北-山东、西南-山东、东北-山东。
   - 输出价差趋势、预测价差点位、区间、跨区经营建议。
   - 同时补充手工/默认运费和净回款价差。

暂未把 95#汽油、柴油作为主预测对象。

## 2. 系统使用的数据

当前数据分为五类。

### 2.1 价格数据

主要来自 Wind 实时接口、ETA 指标库和本地降级快照。

核心价格包括：

- Brent 原油价格，优先使用 Wind 实时价格接口 `GET /price?code=B.IPE&field=rt_latest,rt_chg,rt_pct_chg`
- 山东 92#汽油市场价
- 全国 92#汽油市场价
- 华东 92#汽油市场价
- 华北 92#汽油市场价
- 华南 92#汽油市场价
- 华中 92#汽油市场价
- 西北 92#汽油市场价
- 西南 92#汽油市场价
- 东北 92#汽油市场价

Brent 实时价优先使用 Wind；Wind 不可用时再降级到 ETA；Wind 和 ETA 都不可用时使用本地快照补位。其他成品油价格主要来自 ETA，ETA 不可用时使用本地快照补位，并在页面上标记数据质量。

### 2.2 成本和利润数据

当前系统使用以下成本相关指标：

- Brent 1日、3日变化
- 山东汽油裂解价差
- 山东炼厂利润
- MTBE价格
- 石脑油价格
- 汽油-石脑油价差

这些数据用于判断成本端对山东 92#价格的推涨或压跌作用。

### 2.3 供需数据

当前使用的供需代理变量包括：

- 山东汽油周度销量
- 山东地炼开工或加工量
- 山东汽油出货量
- 利润变化
- 价格均线和短期价格变化

这些变量用于判断山东市场自身供需强弱。

### 2.4 政策数据

系统接入国内成品油调价政策信息，主要用于判断：

- 发改委调价窗口
- 距离下轮调价窗口的工作日
- 最近汽油调价幅度
- 政策窗口对现货报价和库存行为的扰动

如果调价窗口临近，系统会提高政策因子的权重，并在经营建议中提示减少跨窗口囤货。

### 2.5 新闻和事件数据

当前有两类新闻事件：

- 成品油资讯：炼厂、出货、成交、检修、价格变化等。
- 事件风险：地缘、OPEC、库存、制裁、红海、霍尔木兹、供应中断等。

新闻不会直接“让 LLM 猜价格”，而是先进入规则化事件评分，再成为预测因子之一。

## 3. 研判总体流程

系统的价格研判流程如下：

```text
数据采集
  ↓
构造特征
  ↓
各智能体分别打分
  ↓
合成综合分
  ↓
用历史数据校准综合分与价格变化的关系
  ↓
生成趋势、点位、区间和研判可靠度
  ↓
生成核心驱动和经营建议
  ↓
LLM只负责把解释写得更清楚，不改点位和区间
```

关键点：价格点位和区间不是 LLM 随机生成的，而是由确定性打分模型和历史校准结果生成。

## 4. 参与研判的智能体

当前主预测链路中有 8 类子智能体。

### 4.1 业务打分模型智能体

执行《山东成品油市场价预测打分模型.docx》转写后的配置化规则。

当前已经接入代码主链，不再只是方案描述。

代码落点如下：

- 规则配置：`configs/scorecards/shandong_scorecards_v1.yaml`
- 规则执行器：`app/services/scorecard_engine.py`
- 智能体名称：`business_scorecard_agent`
- 主预测入口：`app/services/predictors/shandong_gas92.py`
- 前端和智能体管理展示来源：`agent_claims`、`factor_breakdown`、`raw_context.business_scorecard`

主要看：

- Brent 不同周期涨跌
- 汽油裂解价差趋势
- 山东销量相对强弱
- 市场情绪
- 库存趋势
- 发改委调价窗口预期

输出：

- 业务规则基线分
- 每个规则组的命中分
- 尚未能自动执行的空白阈值，例如原文未明确给出开工率分档时会标记为待业务补齐

当前 92# 预测里，它作为独立业务基准模型单独出分、单独出点位，用来和智能体综合预测对比。它不再作为第一个子智能体混入综合分，也不参与综合可靠度计算。

### 4.2 成本智能体

关注原油和成本变化。

主要看：

- Brent 1日变化
- Brent 3日变化
- 裂解价差
- MTBE、石脑油变化

输出：

- 成本端利多、利空或中性判断
- 对价格方向的贡献分
- 证据说明

### 4.3 市场结构智能体

关注价格结构和短期价格动量。

主要看：

- 山东92#短期价格变化
- 5日、10日均线
- 山东与全国、区域价格关系
- 裂解价差变化

输出：

- 当前价格结构偏强、偏弱或震荡
- 结构性支撑或压力

### 4.4 供给智能体

关注供应端变化。

主要看：

- 山东炼厂开工或加工量
- 出货量
- 利润变化
- 检修、停工、复产类资讯

输出：

- 供应收紧或供应宽松判断

### 4.5 需求智能体

关注终端需求和成交节奏。

主要看：

- 产销率
- 出货强弱
- 价格变化后的成交反馈
- 季节性因素

输出：

- 需求改善、需求走弱或需求平稳判断

### 4.6 成品油资讯智能体

关注国内成品油相关资讯。

主要看：

- 炼厂报价变化
- 地炼出货
- 成交气氛
- 检修复产
- 区域流向

输出：

- 新闻层面对现货价格的方向影响

### 4.7 政策周期智能体

关注发改委调价窗口和政策扰动。

主要看：

- 距离下轮调价窗口的天数
- 最近一次汽油调价幅度
- 调价窗口是否临近

输出：

- 政策窗口对价格的支撑、压制或扰动

### 4.8 事件风险智能体

关注黑天鹅和突发事件。

主要看：

- 地缘冲突
- 制裁
- OPEC
- 红海、霍尔木兹
- 供应中断
- 库存突变
- 物流扰动

输出：

- 短期突发风险强度
- 是否需要触发预警或人工复核

## 5. 智能体如何打分

每个智能体会输出一条判断，包含：

- 方向：up、down、flat
- 研判可靠度：low、medium、high
- 原始分
- 加权分
- 证据
- 结构化信号

示例：

```text
成本智能体：
Brent 1日上涨，裂解价差改善，对山东92#形成正向支撑。
方向：up
贡献分：+1.72
```

系统会把业务基准模型之外的规则智能体按专家权重合成综合分。

```text
综合分 = 成本贡献 + 市场结构贡献 + 供给贡献 + 需求贡献 + 资讯贡献 + 政策贡献
```

综合分越高，系统越倾向于判断价格上行；综合分越低，系统越倾向于判断价格下行；综合分接近 0 时，判断为震荡。

业务打分模型的结果单独展示，不进入上述综合分。

### 5.1 业务打分模型具体怎么执行

业务打分模型不是让 LLM 读 DOCX 后自由判断，而是把 DOCX 中的业务逻辑整理成一套“可执行打分规则配置”，代码按配置逐条计算。

这套配置保存在：

```text
configs/scorecards/shandong_scorecards_v1.yaml
```

它不是原文转存，而是把业务规则拆成以下结构：

```text
scorecard_code：预测对象，例如 sd_gas92
horizons：业务规则周期，例如 D1、W1；系统展示周期为 D1、D3、W1、M1
factor_groups：因子组，例如成本、供给、需求、情绪、库存、政策
features：每个因子组下具体使用的指标
method：每个指标的打分方法
rules：阈值、枚举值和对应分数
score_cap：单个因子组最高/最低贡献限制
unresolved_thresholds：原文没有写清楚、暂不能自动打分的项目
```

当前 `sd_gas92` 已接入的规则组包括：

- 成本：Brent 涨跌、汽油裂解价差趋势。
- 供给：炼厂降负、检修、复产等资讯调整；原 DOCX 中开工率分档阈值未写全，系统会标记为待补齐，不会自行臆造。
- 需求：山东销量相对过去均值的强弱。
- 情绪：成品油资讯中的挺价、让利、推涨、降价等词汇信号。
- 库存和政策：W1 口径下读取库存趋势和调价窗口预期。

举例说明，DOCX 里的“Brent 涨跌对成本端的影响”被整理成了明确分档：

```text
D1 Brent 变动 >= +1.5 美元/桶：+40 分
D1 Brent 变动 +0.5 到 +1.5：+20 分
D1 Brent 变动 -0.5 到 +0.5：0 分
D1 Brent 变动 -1.5 到 -0.5：-20 分
D1 Brent 变动 <= -1.5：-40 分
```

需求侧也不是一句“需求好/需求差”，而是转换成销量相对强弱（产销率）：

```text
sales_ratio >= 120：强利多
105 到 120：利多
100 到 105：小幅利多
95 到 100：中性
85 到 95：小幅利空
70 到 85：利空
低于 70：强利空
```

代码执行时会把配置里的业务字段映射到系统已有特征，例如：

```text
brent_change_usd_d1      -> brent_change_1d
brent_change_usd_w1      -> brent_change_5d
gasoline_crack_trend_d1  -> gasoline_crack_change_3d 后判断扩张/收缩
sales_ratio_d1           -> 山东汽油销量 / 近20日均值
trader_sentiment_label   -> 成品油资讯关键词识别后的情绪标签
price_window_expectation -> 调价窗口天数 + Brent 近3日变化
```

当前使用的打分方法主要有四类：

```text
bucket_score：按数值区间给分，例如 Brent 涨跌、销量强弱。
enum_score：按状态给分，例如裂解价差扩张/持平/收缩。
bounded_numeric：把资讯调整分限制在上下限内，例如炼厂降负资讯 -5 到 +5。
calendar_month_band：按月份识别旺季、淡季、中性月份。
```

如果 DOCX 中某项只有方向描述、没有明确阈值，系统不会补脑。例如山东 CDU 开工率分档目前会进入：

```text
unresolved_items
```

这表示“业务规则已识别，但还缺少可自动执行的分档阈值”，需要业务人员补齐后才能进入正式自动打分。

一次预测输出中可以看到：

```text
business_scorecard_agent
  ↓
scorecard_total：业务规则总分
group_scores：成本、供给、需求、情绪等分组得分
unresolved_items：原规则中缺少明确阈值、暂不能自动执行的项目
```

这样做的目的有两个：

1. 保留业务人员原来的可解释规则。
2. 让规则结果可回放、可复核、可优化，而不是靠模型临场发挥。

## 6. 如何从综合分变成点位

系统不是直接把综合分当成价格涨跌幅。

当前不再使用“截距 + 斜率 × 综合分”的旧线性换算。系统先把智能体综合分换成分数点，再进入智能体综合分专用状态桶：

```text
分数点 = 智能体综合分 × 100
状态桶 = 强空 / 偏空 / 弱空 / 震荡 / 弱多 / 偏多 / 强多
```

然后查找历史上同状态桶样本的真实涨跌分布，取中位数作为基础预测涨跌额：

```text
基础预测涨跌额 = 当前状态桶历史真实涨跌的 P50 中位数
预测点位 = 当前山东92#价格 + 预测变化量
```

业务打分模型也使用“状态桶到历史分布”的框架，但它有自己的业务分状态桶阈值，不和智能体综合分共用。例如业务总分 70 分会进入业务强多桶 `>=70`，而不是套用智能体综合分的 `>=12` 强多桶。

如果当前业务强多桶历史样本不足，系统只允许看相邻桶；仍不足时走专家冷启动，不再用弱多或震荡样本硬凑点位。

## 7. 如何生成区间

预测区间不是固定拍脑袋给出的。

系统会根据两个因素确定区间宽度：

1. 当前状态桶历史涨跌分布的 P25/P75
2. 当前预测周期
3. 数据缺失、因子分歧和事件风险

周期越远，区间通常越宽。

当前最低区间宽度规则：

```text
D1：基础区间较窄
D3/W1/M1：随周期拉长逐步放宽
前端主展示为核心交易区间，半宽封顶为 50 元/吨
如果状态桶分布或风险因子显示波动更大，系统会在 raw_context 中保留风险外沿区间
```

最终输出：

```text
区间半宽 = 状态桶分布半宽 + 数据/分歧/事件风险修正
预测区间 = 预测点位 ± 区间半宽
```

这里的 ±50 不是说模型点位必须精准到 50 元/吨以内，而是把首页展示的“可执行核心区间”收窄，避免业务上看到过宽区间无法决策。完整风险边界保留在：

```text
raw_context.risk_range_lower
raw_context.risk_range_upper
raw_context.risk_range_half_width
```

## 8. 如何判断趋势

系统会比较预测变化量和对应周期的方向阈值。

```text
如果预测变化量 > 阈值：判断上涨
如果预测变化量 < -阈值：判断下跌
如果介于两者之间：判断震荡
```

不同周期使用不同阈值。D1 阈值较小，D3、W1、M1 阈值逐步提高，避免远期预测过度敏感。

## 9. 如何计算研判可靠度

当前系统里的研判可靠度不是“价格一定落在区间内的概率”，而是“本次方向、点位和经营动作可被采纳的可靠度评分”。它按行业专家建议拆成五个分项：

1. 信号强度
   - 预测变化量相对区间和方向阈值越明显，可靠度越高。

2. 因子同向性
   - 成本、供需、政策、资讯、事件等关键因子越同向，可靠度越高。

3. 历史校准质量
   - 样本量越多、历史误差越小，可靠度越高。

4. 数据质量
   - 实时价格、区域价格、政策、资讯未降级时可靠度更高；本地补位时会扣分。

5. 事件稳定性
   - 黑天鹅、地缘、突发装置和物流事件越强，可靠度越低。

输出为：

```text
low / medium / high
```

前端展示为：

```text
低可靠 / 中可靠 / 高可靠 + 百分比
```

系统会在 `raw_context.confidence_components` 中保留分项得分：

```text
signal_strength
factor_alignment
calibration_quality
data_quality
event_stability
```

例如：方向可靠度高，但事件稳定性低时，系统应提示“可参考方向，但不宜重仓执行”。

## 10. 多周期如何处理

系统不是只预测 W+1。

当前支持：

- D1：明日
- D3：三日
- W1：一周
- M1：一月

每个周期都会单独：

1. 计算目标日期
2. 使用对应周期参数
3. 用历史数据校准
4. 输出点位、区间、趋势、研判可靠度

业务打分模型按结构化 YAML 中 D1/D3/W1/M1 对应规则执行，单独输出业务基准点位。它不再使用旧的截距/斜率校准，也不和智能体综合分共用同一套状态桶阈值。

前端会直接显示 D1/D3/W1/M1，不通过日期切换。

## 11. 区域价差如何研判

区域价差预测的核心对象是：

```text
目标区域价格 - 山东价格
```

例如：

```text
华东-山东价差 = 华东92#价格 - 山东92#价格
```

价差为正，说明目标区域高于山东，理论上更利于山东资源外发；价差为负，说明目标区域低于山东，跨区流向吸引力不足。

系统会对每个区域价差分别做预测，逻辑类似山东92#价格预测：

```text
区域价差综合分
  ↓
历史校准
  ↓
预测价差点位
  ↓
预测区间
  ↓
跨区经营建议
```

当前还补充了估算净回款价差：

```text
净回款价差 = 目标区域价格 - 山东价格 - 运费
```

如果净回款价差为正，说明理论上存在跨区流向空间；如果为负，说明扣除运费后跨区吸引力不足。

当前运费支持手工录入并落 PostgreSQL 表 `regional_freight_setting`：

```text
region_code：目标区域编码
region_name：目标区域名称
freight_value：运费，单位元/吨
source_type：manual 表示手工录入，default 表示系统默认估算
updated_by / updated_at：录入人和更新时间
```

只要用户不修改，系统后续预测会一直沿用最近一次手工录入的运费；未录入时才使用默认运费。

注意：当前仅接入运费，后续仍应接入票据、账期、仓储和回款周期。

## 12. 经营建议如何生成

经营建议不是单纯文本摘要。系统先用规则引擎根据预测方向、变化量、调价窗口、事件风险生成动作；如果启用了 LLM，LLM 只负责把建议改写得更清楚、更贴近业务表达，不允许改写方向、点位、区间和触发阈值。

当前建议包含：

- 动作类型：采购、销售、库存、锁价、政策
- 建议动作
- 触发条件
- 价格限制
- 量化建议
- 风险止损
- 负责人角色
- 背后原因

示例：

```text
动作类型：库存
建议动作：以销定采，控制高位库存
触发条件：预测较现货下移 50 元/吨以上
量化建议：库存降至安全线附近
风险止损：政策上调或炼厂挺价时暂停低价去库
负责人角色：经营经理
```

## 13. 预警如何进入研判

预警不直接替代价格预测，但会影响事件风险因子和经营建议。

预警卡片包含：

- 预警等级
- 事件类型
- 影响区域
- 影响品种
- 方向
- 预计影响
- 可靠度
- 状态
- 建议动作
- 预警分

预警状态支持：

- 新触发
- 跟踪中
- 已解除
- 误报

预警状态会被保存，避免同一事件反复打扰。

## 14. LLM 在系统中的作用

LLM 不负责直接生成点位和区间。

当前 LLM 的作用是：

1. 把确定性预测结果改写成更容易读懂的中文解释
2. 整理核心驱动
3. 优化经营建议表达
4. 在对话中回答用户追问

系统对 LLM 有硬约束：

```text
LLM不能改写方向、点位、区间。
点位、区间和趋势始终由确定性预测结果覆盖。
```

行业专家视角下，经营建议应继续升级为“动作卡”而不是自然语言段落，至少包含：

- 适用场景：上涨、下跌、震荡、价差走扩、价差收敛、黑天鹅冲击。
- 触发条件：预测变化量、研判可靠度、库存天数、调价窗口、区域价差、运费、回款周期。
- 执行动作：采购、销售、库存、锁价、跨区流向、夜盘值守。
- 量化边界：目标库存、报价有效期、最大让利、锁单比例、止损点。
- 复核机制：超过价格阈值或重大事件触发时，必须人工确认。

对话回答时，系统会先生成确定性结论：

```text
按 D1 口径，山东92#判断为震荡，点位 7970.97，区间 7945.97~7995.97。
```

然后再拼接 LLM 对原因和经营动作的解释。

## 15. 如何保证同输入同结果

当前系统已经增加 `input_hash`。

系统会把以下内容参与哈希：

- 产品
- 区域
- 周期
- 研判日期
- 场景文本
- 数据上下文
- 智能体综合分
- 业务打分模型命中的规则组和分项得分
- 运行开关
- runtime control

同一输入会得到同一个 `input_hash`，并生成稳定 `run_id`。

这解决了一个关键问题：

```text
两个用户在同一数据快照和同一配置下提问，应得到相同点位和区间。
```

## 16. 当前系统的边界

当前系统已经可以做研究辅助，但还不能完全替代业务人员拍板。

主要边界包括：

1. 内部成交价尚未接入
   - 实际成交价比市场报价更能反映真实经营。

2. 库存、销量、采购计划尚未完全接入
   - 因此建议只能给方向和动作类型，不能精确到真实采购吨数。

3. 运费、票据、账期目前是估算
   - 区域净回款价差仍需业务数据校准。

4. 新闻事件仍需人工复核
   - 黑天鹅和地缘事件可能存在误报或传导不充分。

5. LLM解释依赖模型配置
   - 但点位、区间不依赖 LLM。

6. DOCX 业务规则仍有待业务补齐项
   - 当前代码已经执行可明确落地的规则。
   - 对于原文没有给出明确阈值的项目，例如部分开工率分档，系统会记录在 `unresolved_items` 中，等待业务人员补齐后再进入自动打分。

## 17. 业务人员应该如何使用

建议按以下顺序看系统：

1. 看价格快照
   - Brent、山东92#、全国92#、重点区域价格。

2. 看重点预警
   - 是否有突发事件、政策调价、炼厂扰动、物流扰动。

3. 看晨报
   - 快速了解今天主判断和多周期方向。

4. 看研究台
   - 重点看 D1/D3/W1/M1 点位和区间。

5. 看核心驱动
   - 判断到底是成本、供需、政策、资讯还是事件在主导。

6. 看经营建议
   - 关注动作类型、触发条件、量化建议和风险止损。

7. 看区域价差
   - 判断是否存在跨区流向机会。

8. 必要时向模型对话追问
   - 例如：“如果 Brent 再涨 2 美元，山东92# D1 和 W1 怎么看？”

## 18. 一句话总结

当前系统的价格研判逻辑是：

```text
用 Wind、ETA 和资讯政策数据构造市场特征，
由业务打分模型和多个专业智能体分别判断成本、供需、结构、政策、资讯和事件，
再通过历史校准把综合分转换成趋势、点位和区间，
最后由规则和 LLM 生成可读解释与经营建议。
```

核心原则：

```text
点位和区间由确定性模型生成；
LLM只负责解释，不负责改数；
预警和经营建议必须能被业务人员复核和执行。
```
